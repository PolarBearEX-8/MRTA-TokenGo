# Smart Transit Token Queue

ระบบคิวเสมือนจริงและจองตั๋วล่วงหน้าผ่านแอปพลิเคชัน สำหรับลดการต่อคิวหน้าตู้ซื้อเหรียญโดยสาร MRT

---

## 1. แนวคิดหลักของระบบ

Smart Transit Token Queue คือระบบที่เปลี่ยนการยืนต่อคิวซื้อตั๋วหน้าตู้ ให้ผู้โดยสารสามารถจองตั๋วผ่านแอปพลิเคชันล่วงหน้าได้ เมื่อมาถึงสถานี ผู้โดยสารจะไปรับ Token ที่ตู้ Express Token Machine โดยใช้การยืนยันผ่านแอปและ QR ที่สร้างจากตู้

จุดสำคัญคือ **แอปจะไม่สร้าง QR สำหรับรับ Token โดยตรง** แต่จะออกเป็น **ตั๋วในแอป** เพื่อความปลอดภัย เมื่อผู้โดยสารมาถึงตู้ ตู้จะสร้าง QR ชั่วคราว แล้วให้แอปสแกน QR ของตู้เพื่อยืนยันการรับ Token คล้ายการถอนเงินผ่านแอปธนาคาร

---

## 2. เป้าหมายของระบบ

- ลดเวลาการต่อคิวหน้าตู้จำหน่ายตั๋ว
- ลดความแออัดบริเวณหน้าตู้ในสถานี
- เพิ่มความปลอดภัยในการรับ Token
- ลดความเสี่ยงจากการแคป QR หรือส่งต่อ QR ให้ผู้อื่น
- รองรับกรณีผู้ใช้เลือกสถานีผิด
- มีระบบ Cash Balance สำหรับคืนเงินและชำระส่วนต่าง
- มีระบบจัดการกรณีจองแล้วไม่มารับ Token

---

## 3. ผู้ใช้งานหลัก

### ผู้โดยสาร

ใช้แอปเพื่อเลือกสถานี จองตั๋ว ชำระเงิน และรับ Token ผ่านตู้ Express

### Express Token Machine

ตู้สำหรับรับ Token แบบด่วน โดยทำหน้าที่สร้าง QR ชั่วคราว ตรวจสอบกับ Server และจ่าย Token เมื่อได้รับคำสั่ง

### Server

เป็นตัวกลางตรวจสอบตั๋ว สถานี ค่าโดยสาร สถานะเครื่อง คำสั่งจ่าย Token การคืนเงิน และ Cash Balance

---

## 4. Flow การจองตั๋ว

```text
1. ผู้โดยสารเปิดแอป
        ↓
2. เลือกสถานีต้นทางและปลายทาง
        ↓
3. ระบบคำนวณค่าโดยสาร
        ↓
4. ผู้โดยสารชำระเงิน
        ↓
5. ระบบออก “ตั๋วในแอป”
        ↓
6. ตั๋วมีสถานะ Active และรอการรับ Token
```

หลังชำระเงิน ผู้โดยสารจะไม่ได้รับ QR ทันที แต่จะได้รับตั๋วในแอป เช่น

```text
Ticket ID: STT-10293
Route: Station 2 → Station 3
Fare: 20 บาท
Status: Active
```

---

## 5. Flow การรับ Token ที่ตู้

```text
1. ผู้โดยสารมาถึงสถานี
        ↓
2. ไปที่ตู้ Express Token Machine
        ↓
3. กดปุ่ม “รับ Token จากตั๋วในแอป” บนตู้
        ↓
4. ตู้สร้าง QR ชั่วคราว
        ↓
5. ผู้โดยสารใช้แอปสแกน QR ของตู้
        ↓
6. แอปส่งข้อมูล Ticket + Machine Session ไปที่ Server
        ↓
7. Server ตรวจสอบข้อมูลทั้งหมด
        ↓
8. หากถูกต้อง Server สั่งตู้จ่าย Token
        ↓
9. ตั๋วเปลี่ยนสถานะเป็น Redeemed
```

QR ที่ตู้สร้างเป็นเพียง **Machine Session QR** ไม่ใช่ตั๋วโดยสาร จึงมีอายุสั้น เช่น 30–60 วินาที และใช้เพื่อยืนยันว่าผู้โดยสารอยู่หน้าตู้จริง

---

## 6. เหตุผลที่ให้ตู้สร้าง QR แทนแอป

วิธีเดิมที่ให้แอปสร้าง QR อาจมีความเสี่ยง เช่น

- ผู้ใช้แคปหน้าจอ QR
- ส่ง QR ให้คนอื่น
- QR ถูกขโมยจากมือถือ
- มีการใช้ QR ซ้ำหรือใช้ผิดคน

ระบบใหม่จึงให้ตู้สร้าง QR ชั่วคราว แล้วให้แอปเป็นฝ่ายสแกน QR ของตู้แทน ข้อดีคือ

- QR ไม่ค้างอยู่ในมือถือ
- QR หมดอายุเร็ว
- ต้องมีตั๋วจริงในแอป
- ต้องอยู่หน้าตู้จริง
- Server เป็นผู้อนุมัติทุกครั้งก่อนจ่าย Token

---

## 7. สถานะของตั๋ว

| สถานะ | ความหมาย |
|---|---|
| Active | ตั๋วยังใช้งานได้ รอรับ Token |
| Redeem Locked | ตั๋วอยู่ในช่วงรอ 1 นาทีหลังเปลี่ยนสถานี ยังรับ Token ไม่ได้ |
| Fare Adjustment Required | ต้องชำระส่วนต่างก่อนรับ Token |
| Redeemed | รับ Token สำเร็จแล้ว |
| Expired | ตั๋วหมดอายุ เพราะไม่มารับ Token ภายในเวลาที่กำหนด |
| Refunded to Cash | คืนเงินเข้า Cash Balance แล้ว |
| Refund Requested | ผู้ใช้ยื่นขอคืนเงินจริง |
| Refund Completed | คืนเงินจริงสำเร็จแล้ว |

---

## 8. กรณีจองแล้วไม่มารับ Token

หากผู้โดยสารจองตั๋วและชำระเงินแล้ว แต่ไม่มารับ Token ระบบจะจัดการดังนี้

```text
1. ตั๋วอยู่ในสถานะ Active
        ↓
2. ผู้โดยสารไม่มารับ Token ภายในเวลาที่กำหนด
        ↓
3. ระบบเปลี่ยนสถานะเป็น Expired
        ↓
4. เงินถูกคืนเข้า Cash Balance ในแอป
        ↓
5. หากผู้ใช้ต้องการเงินจริง ต้องยื่น Refund Request
```

### เงื่อนไขที่แนะนำ

- หากไม่รับ Token ภายใน 3–7 วัน ระบบจะ Expire รายการจอง
- เงินจะถูกคืนเข้า Cash Balance ก่อน
- Cash Balance สามารถใช้ซื้อตั๋วครั้งถัดไปหรือชำระส่วนต่างได้
- หากต้องการถอนเป็นเงินจริง ต้องยื่นคำขอ Refund ผ่านแอป

---

## 9. Cash Balance

Cash Balance คือยอดเงินคงเหลือในแอป ใช้สำหรับ

- รับเงินคืนจากตั๋วที่ Expired
- ชำระส่วนต่างเมื่อเปลี่ยนสถานีแล้วค่าโดยสารสูงขึ้น
- ใช้ซื้อตั๋วครั้งถัดไป
- รอการถอนเงินจริงผ่าน Refund Request

ตัวอย่างรายการ Cash Balance

```text
Booking #STT-10293
Status: Expired
Refunded to Cash Balance: 28 บาท
```

---

## 10. กรณีเลือกสถานีต้นทางผิด

ปัญหานี้เกิดขึ้นเมื่อผู้โดยสารเลือกสถานีต้นทางในแอปไม่ตรงกับสถานีที่มารับ Token จริง เช่น

```text
ผู้ใช้จอง: Station 2 → Station 3
ผู้ใช้อยู่จริง: Station 1
```

เมื่อผู้โดยสารใช้แอปสแกน QR ที่ตู้ Server จะตรวจสอบสถานีของตู้ หากพบว่าไม่ตรงกับตั๋ว ระบบจะไม่จ่าย Token ทันที แต่จะแจ้งให้ผู้โดยสารเปลี่ยนสถานีต้นทาง

---

## 11. การเปลี่ยนสถานีต้นทาง

ระบบอนุญาตให้เปลี่ยนสถานีต้นทางได้ แต่ต้องอยู่ภายใต้เงื่อนไขเพื่อป้องกันการใช้งานผิดวัตถุประสงค์

### เงื่อนไข

- เปลี่ยนได้เฉพาะก่อนรับ Token
- เปลี่ยนได้เฉพาะสถานีต้นทาง
- ปลายทางยังคงเดิม
- เปลี่ยนฟรีได้ 2 ครั้งต่อ 1 รายการจอง
- หลังเปลี่ยนสถานีแต่ละครั้ง จะ Redeem หรือรับ Token ไม่ได้ 1 นาที
- ครั้งที่ 3 จะเปลี่ยนฟรีไม่ได้ ต้องจ่ายค่าธรรมเนียม หรือระบบอาจไม่อนุญาตให้เปลี่ยน
- หากรับ Token แล้ว สถานะเป็น Redeemed และเปลี่ยนสถานีไม่ได้อีก

---

## 12. Flow การเปลี่ยนสถานี

```text
1. ผู้โดยสารสแกน QR จากตู้
        ↓
2. Server ตรวจสอบสถานีของตู้
        ↓
3. พบว่าสถานีไม่ตรงกับตั๋ว
        ↓
4. แอปแจ้งเตือนให้เปลี่ยนสถานีต้นทาง
        ↓
5. ระบบคำนวณค่าโดยสารใหม่
        ↓
6. ผู้โดยสารกดยืนยันเปลี่ยนสถานี
        ↓
7. ระบบตรวจสอบจำนวนครั้งที่เปลี่ยน
        ↓
8. หากยังไม่เกิน 2 ครั้ง เปลี่ยนฟรี
        ↓
9. ตั๋วถูกล็อกไม่ให้ Redeem 1 นาที
        ↓
10. เมื่อครบเวลา ผู้โดยสารจึงรับ Token ได้
```

---

## 13. กรณีค่าโดยสารเปลี่ยนหลังเปลี่ยนสถานี

### 13.1 ค่าโดยสารแพงขึ้น

หากค่าโดยสารใหม่สูงกว่าราคาที่จ่ายไว้ ผู้โดยสารต้องจ่ายส่วนต่างก่อนรับ Token

```text
ค่าโดยสารเดิม: 20 บาท
ค่าโดยสารใหม่: 28 บาท
ต้องจ่ายเพิ่ม: 8 บาท
```

ช่องทางจ่ายส่วนต่าง

- ใช้ Cash Balance
- ชำระผ่านช่องทางในแอป เช่น PromptPay, e-Wallet หรือบัตร
- ใช้ Cash Balance บางส่วน และจ่ายส่วนที่เหลือผ่านช่องทางอื่น

### 13.2 ค่าโดยสารเท่าเดิม

ระบบเปลี่ยนสถานีให้ทันที แต่ยังต้องรอ 1 นาทีตามกฎ Redeem Lock

### 13.3 ค่าโดยสารถูกลง

ระบบจะไม่คืนส่วนต่าง แต่จะให้ผู้ใช้ยืนยันว่าจะใช้ตั๋วเดิมต่อ

```text
ค่าโดยสารเดิม: 28 บาท
ค่าโดยสารใหม่: 20 บาท
ระบบจะไม่คืนส่วนต่าง
```

หากผู้ใช้ยืนยัน ระบบจะเปลี่ยนสถานี และล็อกการรับ Token 1 นาที

---

## 14. กรณีเปลี่ยนสถานีครั้งที่ 3

เมื่อผู้โดยสารเปลี่ยนสถานีครบ 2 ครั้งแล้ว ครั้งที่ 3 จะไม่สามารถเปลี่ยนฟรีได้

แนวทางที่แนะนำคือ **อนุญาตให้เปลี่ยนได้ แต่ต้องเสียค่าธรรมเนียม** เพราะยืดหยุ่นกว่าการบล็อกผู้ใช้ทันที

```text
คุณเปลี่ยนสถานีฟรีครบ 2 ครั้งแล้ว
หากต้องการเปลี่ยนอีกครั้ง ต้องชำระค่าธรรมเนียมเพิ่มเติม
```

ระบบสามารถให้จ่ายค่าธรรมเนียมผ่าน

- Cash Balance
- ช่องทางชำระเงินในแอป

---

## 15. Virtual Queue

หากมีผู้โดยสารมารับ Token ที่ตู้ Express จำนวนมาก ระบบสามารถใช้ Virtual Queue ได้

```text
1. ผู้โดยสารเปิดหน้ารับ Token
        ↓
2. ระบบตรวจสอบจำนวนคนหน้าตู้
        ↓
3. หากคนเยอะ ระบบออกคิวเสมือน
        ↓
4. แอปแจ้งว่าเหลืออีกกี่คิว
        ↓
5. เมื่อถึงคิว ผู้โดยสารจึงไปสแกน QR ที่ตู้
```

ข้อดี

- ลดการยืนรอหน้าตู้
- ลดความแออัดในสถานี
- ทำให้ผู้โดยสารรู้เวลารอโดยประมาณ

---

## 16. ความปลอดภัยของระบบ

ระบบควรมีมาตรการดังนี้

- ตั๋วในแอปใช้แทน QR ถาวร
- QR ถูกสร้างจากตู้และมีอายุสั้น
- QR ของตู้ต้องตรวจสอบกับ Server ทุกครั้ง
- Ticket ID ต้องใช้ได้ครั้งเดียว
- หลังรับ Token สำเร็จ สถานะต้องเป็น Redeemed ทันที
- ตู้ต้องไม่จ่าย Token หาก Server ไม่อนุมัติ
- ทุกเหตุการณ์ต้องมี Log เช่น เวลา สถานี หมายเลขตู้ และสถานะตั๋ว
- หากตู้จ่าย Token ไม่สำเร็จ ต้องไม่เปลี่ยนสถานะเป็น Redeemed

---

## 17. ปัญหาที่ระบบต้องรองรับ

| ปัญหา | แนวทางแก้ |
|---|---|
| จองแล้วไม่มารับ Token | Expire และคืนเงินเข้า Cash Balance |
| ต้องการเงินจริงคืน | ยื่น Refund Request |
| เลือกสถานีผิด | ให้เปลี่ยนสถานีต้นทางก่อนรับ Token |
| ค่าโดยสารแพงขึ้น | จ่ายส่วนต่างก่อนรับ Token |
| ค่าโดยสารถูกลง | ไม่คืนส่วนต่าง แต่ใช้ตั๋วต่อได้ |
| เปลี่ยนสถานีบ่อยเกินไป | ฟรี 2 ครั้ง ครั้งที่ 3 เสียค่าธรรมเนียมหรือเปลี่ยนไม่ได้ |
| เปลี่ยนสถานีแล้วรีบ Redeem | ล็อกการ Redeem 1 นาที |
| QR ถูกแคปหรือส่งต่อ | แอปไม่มี QR ถาวร ตู้เป็นฝ่ายสร้าง QR ชั่วคราว |
| ตู้เสียหรือจ่าย Token ไม่ออก | สถานะเป็น Dispense Failed และให้ลองตู้ใหม่หรือแจ้งเจ้าหน้าที่ |

## 18. Prototype Scope

Prototype ควรแสดงหน้าหลักดังนี้

1. หน้าเลือกสถานีต้นทางและปลายทาง
2. หน้าชำระเงินและสร้างตั๋วในแอป
3. หน้าตั๋วในแอป
4. หน้าจำลองตู้ Express Token Machine
5. หน้าสแกน QR จากตู้
6. หน้าตรวจสอบสถานีไม่ตรงกัน
7. หน้าชำระส่วนต่างด้วย Cash Balance
8. หน้าล็อก 1 นาทีหลังเปลี่ยนสถานี
9. หน้ารับ Token สำเร็จ
10. หน้า Cash Balance และ Refund Request

---

## 19. Logic สำคัญของระบบ

```text
ถ้าสถานีตรงกัน
    → ตรวจสอบสถานะตั๋ว
    → ถ้า Active และไม่ติด Lock
    → สั่งตู้จ่าย Token

ถ้าสถานีไม่ตรงกัน
    → แจ้งผู้ใช้
    → ให้เปลี่ยนสถานีต้นทาง
    → คำนวณค่าโดยสารใหม่
    → ถ้าต้องจ่ายเพิ่ม ให้ชำระส่วนต่าง
    → ถ้าราคาถูกลง ไม่คืนส่วนต่าง
    → ล็อก Redeem 1 นาที
    → ครบเวลาแล้วจึงรับ Token ได้

ถ้าเปลี่ยนสถานีครบ 2 ครั้งแล้ว
    → ครั้งต่อไปต้องเสียค่าธรรมเนียม หรือไม่อนุญาตให้เปลี่ยน

ถ้าไม่มารับ Token ภายใน 3–7 วัน
    → Expire
    → คืนเงินเข้า Cash Balance
    → หากต้องการเงินจริง ให้ยื่น Refund Request
```
